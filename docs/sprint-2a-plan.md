# Sprint 2A — Reliability Core: Operational Plan

Convert Librarian from experimental alpha to a review-driven, deterministic, recoverable runtime.

Last updated: 2026-05-13

---

## Architectural Rules

These are non-negotiable constraints for Sprint 2A:

1. **No full autonomy.** All operations remain manually triggered.
2. **No filesystem watchers.** They multiply race conditions, invalidation bugs, partial rebuilds, duplicate events, and debounce complexity. The transactional base must be solid before adding reactivity.
3. **No automatic loops.** No background agents, auto-apply, or autonomous reconciliation.
4. **Review-driven model preserved.** `.librarian/proposals/` remains the source of truth. `reviews/` remains the human-readable export surface.
5. **Every mutation is deterministic and recoverable.** Given the same inputs, the same outputs. Given a failure, a clear path to recovery.
6. **`operationId` propagates through all mutation paths.** Every apply, rebuild, retry, and recovery operation generates a unique correlation ID that flows through proposals, transactions, ledger, logs, and TUI.

---

## Bounded Contexts

Sprint 2A formalizes two separate bounded contexts that must never be conflated:

### Processed Ledger

Answers: **"Did we already ingest this source?"**

- Location: `.librarian/processed.json`
- Key: source path
- States: `processed` | `skipped` | `errored`
- Written: **only after successful apply completes entirely**
- Contains: `operationId` of the apply that processed it

### Proposal Lifecycle

Answers: **"What happened operationally with this proposal?"**

- Location: `.librarian/proposals/<id>/metadata.json`
- Key: proposal id
- States: `pending` → `applying` → `applied` | `failed` | `rolled_back`
- Written: throughout the entire proposal lifecycle
- Contains: full transition history with `operationId` per attempt

**The only coupling point:** the apply pipeline writes to both — ledger last, after everything else succeeds.

## 1. Proposal State Model

### State Machine

```text
pending ──→ applying ──→ applied
   ↑            │
   │            ↓
   │         failed
   │            │
   │            ↓
   └─── rolled_back
```

### State Definitions

| State | Meaning | Next Allowed | Entry Criteria |
|-------|---------|-------------|----------------|
| `pending` | Created, awaiting approval | `applying` | Proposal generated by Librarian |
| `applying` | Apply in progress | `applied`, `failed`, `rolled_back` | User approved + apply started |
| `applied` | Successfully applied | Terminal | All targets written + verified |
| `failed` | Apply failed, partial writes may exist | `applying` (retry) | Error during apply |
| `rolled_back` | Apply failed + rollback completed | `applying` (retry) | Error + rollback succeeded |

### Transitions are:

- **Validated** — only allowed transitions are accepted
- **Persisted** — each transition is recorded with timestamp, operationId, and attempt number
- **Auditable** — full history is retained, never deleted

### Metadata Schema

`.librarian/proposals/<id>/metadata.json`:

```json
{
  "id": "prop_abc123",
  "state": "pending",
  "createdAt": "2026-05-13T10:00:00Z",
  "updatedAt": "2026-05-13T10:00:00Z",
  "appliedAt": null,
  "sourceProcessedAt": null,
  "attempts": 0,
  "lastError": null,
  "targets": [
    { "path": "wiki/conceptos/foo.md", "action": "create" },
    { "path": "wiki/index.md", "action": "update" }
  ],
  "transitions": [
    {
      "operationId": "op_01H...",
      "from": "pending",
      "to": "applying",
      "at": "2026-05-13T10:00:00Z",
      "attempt": 1,
      "reason": "apply-start"
    }
  ]
}
```

## 2. Transaction-Like Apply

### Core Principle

The apply operation is the most critical mutation in the system. It must be:

- **Atomic in intent** — either all targets are written, or none persist
- **Recoverable** — partial state is detectable and fixable
- **Idempotent for retry** — retrying a failed/rolled_back proposal is safe

### Apply Flow

```text
1. Load proposal from .librarian/proposals/<id>/metadata.json
2. Validate state is in {pending, failed, rolled_back}
3. Generate operationId: op_<ulid>
4. Create transaction record in .librarian/transactions/<operationId>.json
5. Transition proposal: current → applying (persist)
6. For each target:
   a. Write to temp file (same directory as final target)
   b. Validate written content (size > 0, parseable if applicable)
   c. Atomic rename: temp → final target
   d. Record target completion in transaction
7. Update wiki/index.md and wiki/log.md
8. Update processed ledger entry for source (with operationId)
9. Transition proposal: applying → applied (persist)
10. Mark transaction: completed
```

### Failure Flow

```text
1. Capture error
2. Record error in transaction (with stack/cause summary)
3. For each already-written target:
   a. If previous content was backed up → restore from backup
   b. If new file → delete
4. If rollback completed fully:
   → transition: applying → rolled_back
5. If rollback partially failed:
   → transition: applying → failed (with rollbackError)
6. Persist lastError in proposal metadata
7. Do NOT update processed ledger
8. Do NOT mark source as processed
```

### Critical Ordering Rules

1. **Processed ledger is written LAST.** Never before all targets are verified.
2. **Source is never marked processed before apply succeeds.**
3. **Each attempt gets a new operationId.** Retries are traceable.
4. **Temp files use the same filesystem as the target.** This ensures atomic rename works (same mount point).

### Transaction Record Schema

`.librarian/transactions/<operationId>.json`:

```json
{
  "operationId": "op_01H...",
  "proposalId": "prop_abc123",
  "attempt": 2,
  "startedAt": "2026-05-13T10:00:00Z",
  "completedAt": null,
  "status": "in_progress",
  "targets": [
    {
      "path": "wiki/conceptos/foo.md",
      "action": "create",
      "status": "completed",
      "tempPath": "wiki/conceptos/foo.md.tmp_op_01H",
      "previousHash": null,
      "completedAt": "2026-05-13T10:00:01Z"
    }
  ],
  "error": null,
  "rollbackError": null
}
```

### Required Tests

| Test | Validates |
|------|-----------|
| Happy path apply | All targets written, proposal → applied, ledger updated |
| Fail writing temp target | Proposal → failed, no final targets written, ledger unchanged |
| Fail after temp write, before rename | Proposal → failed/rolled_back, temp cleaned up, ledger unchanged |
| Fail after partial multi-target apply | Rolled back targets restored, failed targets unchanged, ledger unchanged |
| Rollback restores previous content | Previous file content is recovered after failed apply |
| Retry on failed proposal | New operationId, attempt incremented, no duplicate writes |
| Retry on applied proposal | Rejected — applied is terminal |
| Source ledger not changed on failure | Ledger entry unchanged after any failure scenario |

## 3. Proposal Inspection and Recovery

### CLI Commands

```bash
librarian proposal <id>
```

Output:

```text
Proposal: prop_abc123
State:    failed
Created:  2026-05-13T10:00:00Z
Updated:  2026-05-13T10:00:05Z
Attempts: 2
Last error: EACCES: permission denied, open 'wiki/conceptos/foo.md'

Transitions:
  1. pending → applying   (op_01H..., attempt 1) at 10:00:00
  2. applying → failed    (op_01H..., attempt 1) at 10:00:03 — EACCES
  3. failed → applying    (op_01J..., attempt 2) at 10:05:00
  4. applying → failed    (op_01J..., attempt 2) at 10:05:02 — EACCES

Targets:
  - wiki/conceptos/foo.md (create) — not written
  - wiki/index.md (update) — not written
```

```bash
librarian proposal retry <id>
```

- Allowed only for `failed` and `rolled_back`
- For `applying` state: requires explicit `--force` flag after stuck detection (proposal in `applying` for > N minutes)
- Generates new operationId
- Increments attempt counter
- Reuses the same apply pipeline

```bash
librarian proposal reset <id>
```

- Transitions back to `pending`
- Persists transition with audit trail
- Moves `lastError` to transition history
- **Not allowed for `applied` proposals in Sprint 2A** (applied is terminal)

### TUI Integration

- List proposals with `failed` state prominently
- Detect `applying` proposals older than a threshold as `stuck`
- Actions available: view details, retry, reset
- Display: operationId, attempt count, last error, transition timeline

## 4. Index Invalidation and Rebuild

### Command

```bash
librarian index rebuild
```

### Index Metadata Schema

`.librarian/index-metadata.json`:

```json
{
  "builtAt": "2026-05-13T10:00:00Z",
  "invalidatedAt": null,
  "status": "fresh",
  "indexedFiles": {
    "wiki/conceptos/foo.md": {
      "mtimeMs": 1715590800000,
      "size": 2048,
      "hash": "sha256:abc123..."
    }
  },
  "caches": {
    "semantic": { "builtAt": "2026-05-13T10:00:00Z", "status": "fresh" },
    "wikilinks": { "builtAt": "2026-05-13T10:00:00Z", "status": "fresh" },
    "backlinks": { "builtAt": "2026-05-13T10:00:00Z", "status": "fresh" },
    "orphans": { "builtAt": "2026-05-13T10:00:00Z", "status": "fresh" }
  },
  "stats": {
    "totalFiles": 42,
    "totalSize": 128000
  }
}
```

### Invalidation Triggers

Check runs when a command needs the index or when TUI starts. Not continuous.

| Trigger | What invalidates |
|---------|-----------------|
| File path added or removed | wikilinks, backlinks, orphans |
| File `mtimeMs` changed | semantic, wikilinks |
| File `size` changed | semantic, wikilinks |
| File content hash changed | semantic, wikilinks, backlinks |
| Structural folder change | all caches |

### Status Values

| Status | Meaning |
|--------|---------|
| `fresh` | Index reflects current vault state |
| `stale` | Detected changes since last build; operations may warn |
| `missing` | No index exists yet; rebuild required |
| `rebuilding` | Rebuild in progress |

### Rebuild Flow

```text
1. Set status → rebuilding
2. Walk wiki/ and raw/ directories
3. For each file: record mtime, size, hash
4. Rebuild wikilinks graph
5. Rebuild backlinks from wikilinks
6. Rebuild orphan list from backlinks
7. Invalidate semantic cache
8. Set status → fresh
9. Set builtAt → now
10. Set invalidatedAt → null
```

### TUI Display

- Last index rebuild timestamp
- Current index status badge
- Warning when status is `stale`

### Required Tests

| Test | Validates |
|------|-----------|
| Rebuild creates complete metadata | All files indexed, all caches fresh |
| Content change invalidates index | status → stale after file modification |
| Path addition/removal invalidates | status → stale after file add/remove |
| Rebuild clears invalidatedAt | Fresh state after rebuild |
| Stale index warns but doesn't block | Operations proceed with warning |

### Out of Scope

- Filesystem watchers
- Realtime sync
- Incremental partial rebuilds (full rebuild only in 2A)

## 5. /orphans Correctness

### Rule

`/orphans` does NOT walk the filesystem manually. It uses only the graph index and queryApi.

### Source of Truth

- Backlinks graph from index metadata
- `queryApi` for structured queries
- Official graph index

### Exclusions

Pages in these states/locations must NOT appear as orphans:

| Exclude | Reason |
|---------|--------|
| Files in `.librarian/**` | Internal state, not wiki content |
| Files in `reviews/**` | Proposal export surface, not wiki pages |
| Proposal targets not yet applied | Pending state — not real pages yet |
| Temp files (`*.tmp_*`) | Transaction artifacts |
| Partial writes | Incomplete mutations |
| Files outside the wiki graph | Not part of the knowledge layer |

### Required Tests

| Test | Validates |
|------|-----------|
| Real orphan (no backlinks) | Appears in results |
| Connected page (has backlinks) | Does NOT appear |
| Pending proposal target | Does NOT appear |
| Temp file | Does NOT appear |
| After backlink rebuild | Results reflect updated graph |

## 6. Contextual TUI Alignment

### Problem

The TUI chat path may diverge from the actual vault context, using different sources, tools, or restrictions than the CLI/harness path.

### Solution

TUI must be a surface of the real runtime, not a separate chatbot.

### What TUI Reuses

| Component | Source |
|-----------|--------|
| Vault context | `vault-context.ts` |
| Librarian runtime | `runLibrarian` |
| Query API | `queryApi` |
| Graph index | Same index as CLI |
| Proposal state | Same proposal store as CLI |

### What TUI Avoids

- Free-form chat without structured context
- Independent source selection
- Separate tool definitions
- Bypassing proposal/review flow

### Expected Behavior

- Same question via CLI and TUI uses the same sources
- TUI respects pending/failed proposals before executing actions
- TUI does not respond using context outside vault-context
- TUI warns if index is stale
- TUI shows proposal state, index status, and recovery options

### Suggested Module

Extract a shared `runtime-context.ts` if CLI and TUI currently duplicate logic:

```text
src/
  runtime/
    runtime-context.ts      ← shared entry point
    proposal-store.ts       ← read/write proposals + metadata
    proposal-state.ts       ← validate transitions
    proposal-apply.ts       ← transaction-like apply
    transaction-store.ts    ← transaction records
    index-store.ts          ← index metadata + invalidation
    index-rebuild.ts        ← full rebuild
    processed-ledger.ts     ← separate bounded context
    graph-query.ts          ← official source for orphans/links
```

## operationId Propagation

Every mutation operation generates a unique `operationId` (`op_` prefix). It propagates to:

| Consumer | Field |
|----------|-------|
| Proposal metadata | `transitions[].operationId` |
| Transaction record | `operationId` |
| Processed ledger | `operationId` |
| Log lines | `operationId` field in canonical log line |
| TUI status | Displayed in proposal detail view |

Each retry attempt = new operationId. This enables:

- Tracing a specific retry in logs
- Correlating TUI display with backend state
- Debugging partial failures across systems

## Processed Ledger Schema (Separate Bounded Context)

`.librarian/processed.json`:

```json
{
  "entries": {
    "raw/articles/some-article.md": {
      "status": "processed",
      "operationId": "op_01H...",
      "proposalId": "prop_abc123",
      "processedAt": "2026-05-13T10:00:05Z",
      "hash": "sha256:def456..."
    }
  }
}
```

Key properties:

- Keyed by source path, not by proposal id
- Independent of proposal lifecycle
- Written only after successful apply
- Contains operationId for traceability

These are explicitly NOT part of Sprint 2A. They belong to Sprint 2B+:

- Autonomous reconciliation
- Multi-agent orchestration

## Acceptance Criteria

Sprint 2A is complete when:

- [ ] Apply is idempotent for safe retry
- [ ] Source processed ledger only changes after complete success
- [ ] Every proposal has persisted, observable state with full transition history
- [ ] Failed/stuck proposals are visible via CLI and TUI
- [ ] `librarian index rebuild` exists and updates metadata
- [ ] `/orphans` depends on graph index, not filesystem walking
- [ ] TUI uses the same runtime context as CLI/harness
- [ ] operationId propagates through all mutation paths
- [ ] Processed ledger and proposal lifecycle are separate stores
- [ ] No watchers, loops, auto-apply, or autonomy added
